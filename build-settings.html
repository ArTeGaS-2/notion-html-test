Дві «реальності» гри

У Unity є дві «реальності» твоєї гри. Перша – це проєкт у редакторі: сцени, префаби, скрипти, інспектор. Друга – це зібрана гра, яку можна запустити без Unity: .exe на Windows або .apk/.aab на Android. Процес переходу від першої до другої називається побудова (build), а основне місце керування цим процесом – вікно Build Settings.

Вибір платформи та Scenes in Build

У Build Settings ти обираєш платформу, для якої зараз збираєш гру. Для Windows це зазвичай PC, Mac & Linux Standalone з Target Platform = Windows, для мобільних – Android. Unity бере твій проєкт, компілює скрипти, пакує сцени й ресурси у формат, який вміє запускати вибрана платформа. Важливий момент – список Scenes in Build: лише ті сцени, що додані туди, потраплять у фінальну гру. Саме з першої сцени в цьому списку стартує проєкт після запуску .exe або встановлення .apk.

Бекенди скриптів: Mono vs IL2CPP

Окремий шар цієї теми – як саме Unity компілює C#-скрипти. Для цього є два варіанти: Mono і IL2CPP. Їх можна сприймати як два різних «двигуни» для твого C# коду. Mono – це інтерпретатор/віртуальна машина: C# код компілюється у проміжний байт-код, а вже під час запуску гри він додатково перекладається в машинні інструкції. Плюс Mono у тому, що він швидко збирається: Dev-білди на ньому робляться помітно швидше, простіше дебажити, швидко тестувати зміни. Мінус – дещо гірша продуктивність і безпека порівняно з IL2CPP, особливо на мобільних платформах.

IL2CPP для релізів, Mono для швидких збірок

IL2CPP працює інакше: при побудові він перетворює весь твій C# код спочатку в C++, а потім у «рідний» машинний код платформи. Це займає більше часу на етапі збірки, але натомість дає кращу швидкодію і зазвичай менше проблем на реальних пристроях. Для серйозних Android-релізів зараз це фактично стандарт: гра працює швидше, складніше «розібрати» код, менше сюрпризів з різними моделями телефонів. Для навчальних і внутрішніх збірок на Windows часто використовують Mono, щоб не чекати довгі білди, а ближче до фіналу перемикаються на IL2CPP і проганяють уже релізну збірку.

Development Build vs Release

Ще один важливий перемикач у Build Settings – це галочка Development Build. Вона відрізняє умовний «Dev» від «Release». Development Build додає в гру додаткову налагоджувальну інформацію: розширені логи, можливість підключити Profiler, Script Debugging, іноді вмикаються різні перевірки, які допомагають знаходити помилки. Такі збірки зручні для тестів на власному ПК чи телефоні, але вони зазвичай працюють трохи повільніше і можуть важити більше. Release-білд – це те, що отримує гравець: галочка Development Build знята, дебаг-інформація та зайві перевірки вимкнені, логи мінімізовані, часто використовується сильніше стиснення ресурсів. Ідея проста: поки ти ще активно змінюєш код і шукаєш баги – збираєш Dev; коли готуєшся викладати гру на itch.io, Google Play чи просто віддавати друзям – робиш Release.

Результат побудови: Windows і Android

Для Windows результатом побудови стає папка з .exe файлом і підпапкою з усіма даними гри. Користувач просто запускає .exe і грає – Unity йому вже не потрібна. Для Android результат – .apk або .aab. .apk можна одразу встановити на телефон, підключений по USB, або відправити комусь. .aab частіше використовується для публікації в Google Play. У всіх випадках логіка одна: ти один раз налаштовуєш Build Settings, обираєш потрібний бекенд скриптів (Mono чи IL2CPP) і тип білда (Dev/Release), а далі Unity сама перетворює твій універсальний C# код і сцени в формат, який розуміє конкретна платформа.
